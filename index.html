Interfaces.py ABC

Определяет два абстрактных базовых класса — IInputProvider и IRenderer. Они не содержат логики, только контракты (сигнатуры). Любой конкретный класс, реализующий их, обязан предоставить все методы. Это делает Engine полностью независимым от pygame или любого другого фреймворка.

IInputProvider.poll_events(self)
→ str▼

Опрашивает очередь событий операционной системы за текущий кадр. Должен вернуть одну из трёх строк-директив: "quit" — завершить приложение, "reset" — перезапустить игру, "continue" — ничего не делать. Вызывается один раз в начале каждой итерации игрового цикла.

IInputProvider.get_actions(self)
→ tuple[dict, dict]▼

Считывает мгновенное состояние устройств ввода и возвращает пару словарей — по одному на каждого игрока. Каждый словарь содержит ключи "forward", "brake", "left", "right", "nitro" со значениями bool. Игрок 1 — WASD+Shift, Игрок 2 — стрелки+Enter.

Ключ словаря Тип Значение
forward bool Нажата кнопка газа — ускорение вперёд
brake bool Нажата кнопка тормоза — отрицательная тяга
left bool Руль влево — отрицательный поворот
right bool Руль вправо — положительный поворот
nitro bool Нитро — дополнительная тяга ×2.5, удвоение max_speed
IRenderer.setup(self, game)
→ None▼

Вызывается один раз перед стартом игрового цикла. Конкретная реализация использует его для дорогостоящей инициализации: создание окна, генерация поверхности трассы, передача коллбэка проверки стен в Game. Принимает объект Game для настройки взаимодействия.

IRenderer.render(self, game)
→ None▼

Отрисовывает текущее состояние игры. Вызывается один раз за кадр после game.update(). Реализация читает позиции, скорости и счётчики кругов из объектов игры, но никогда не изменяет их — это строгое разделение логики и представления.

IRenderer.tick(self)
→ float▼

Ограничивает FPS и возвращает delta time — время в секундах, прошедшее с предыдущего кадра. Это значение передаётся в Car.update(dt) для фреймрейт-независимой физики. При 60 FPS dt ≈ 0.0167 секунды.

MathVector2 Car.py

Двумерный математический вектор с компонентами x и y типа float. Используется для хранения позиции и скорости машины. Намеренно не использует pygame.Vector2, чтобы модуль Car не зависел от pygame.

length(self)
→ float▼

Возвращает евклидову длину вектора по формуле √(x²+y²). Реализовано через math.hypot(x, y) — это точнее и быстрее, чем sqrt(x*x + y*y), так как hypot защищён от переполнения при очень больших значениях. Используется в Car для измерения скорости.

scale_to_length(self, target_length)
→ None▼

Нормализует вектор (делает длину = 1) и умножает на target_length. Вызывается в Car.update() для ограничения скорости: если vel.length() > max_speed, вектор скорости "обрезается" до максимума без изменения направления движения. Пропускает деление на ноль через проверку if l > 0.

lerp(self, other, t)
→ MathVector2▼

Линейная интерполяция между текущим вектором и other с коэффициентом t ∈ [0.0, 1.0]. Формула: self + (other - self) * t. Применяется для имитации сопротивления воздуха: каждый кадр вектор скорости интерполируется к нулю с коэффициентом DRAG * dt * 0.5.

Параметр Тип Описание
other MathVector2 Целевой вектор. При вызове из Car всегда передаётся MathVector2(0,0) — ноль
t float Коэффициент: 0.0 = без изменений, 1.0 = полный переход к other
__add__, __sub__, __mul__, __rmul__
→ MathVector2▼

Перегрузка операторов для читаемой математики: pos += vel * dt вместо pos.x += vel.x * dt; pos.y += vel.y * dt. __rmul__ нужен для случая когда скаляр стоит слева: 2.5 * vector — Python сначала пробует float.__mul__(vector), это не работает, и тогда вызывает vector.__rmul__(2.5).

Car Car.py

Физическая модель автомобиля. Хранит позицию, вектор скорости и угол направления. Содержит только чистую математику — никакого pygame, никакого рендера. Константы класса (ACCELERATION, DRAG и др.) намеренно вынесены на уровень класса, а не экземпляра — их удобно менять для настройки баланса.

update(self, dt, actions)
→ None▼

Главный физический тик. За каждый кадр выполняет пять шагов по порядку:

1. Поворот — вычисляет знак поворота по клавишам left/right. При скорости менее 10 px/s поворот ослаблен пропорционально скорости — нельзя крутить руль на месте.
2. Тяга — складывает thrust из газа (+1.0), тормоза (−0.5) и нитро (+2.5). При нитро удваивается max_speed.
3. Ускорение — добавляет к вектору скорости forward * ACCELERATION * thrust * dt, где forward — единичный вектор текущего направления машины.
4. Ограничение — если vel.length() > max_speed, скорость обрезается до максимума через scale_to_length.
5. Торможение — применяет lerp к нулю с коэффициентом DRAG * dt * 0.5 — имитация аэродинамического сопротивления.

__copy__ / __deepcopy__
→ raises TypeError▼

Блокировка копирования — питонический аналог = delete из C++. Если где-то в коде случайно напишут copy.copy(car) или copy.deepcopy(car), вместо создания скрытого дубля с независимым состоянием будет немедленно брошено TypeError. Это предотвращает труднообнаруживаемые баги, когда изменения одной машины не отражаются на другой.

PlayerState Game.py

Агрегирует всё состояние одного игрока: его машину, счётчик кругов и флаг отдалённости от финишной линии. Это решает проблему предыдущей версии, где Game хранил отдельные поля laps1, laps2, _p1_away, _p2_away и обращался к ним через getattr/setattr со строковыми именами.

check_lap(self, is_finish)
→ None▼

Принимает функцию-предикат is_finish(x, y) → bool и проверяет, находится ли машина в зоне финиша. Засчитывает круг только если: машина сейчас в зоне финиша (near=True) И флаг _away=True (т.е. она до этого отъехала) И скорость больше 10 px/s (нельзя стоять на финише и получать круги). После проверки обновляет флаг _away = not near.

Параметр Тип Описание
is_finish Callable[[float,float], bool] Коллбэк из Game. Возвращает True если координата (x,y) попадает в зону финишной линии

Game Game.py

Управляет игровым миром. Содержит двух игроков (PlayerState), координаты финишной линии и коллбэк проверки стен. Ключевое улучшение архитектуры: Game не хранит pygame.Surface — вместо этого PyGameRenderer.setup() передаёт в него замыкание is_wall(x, y) → bool. Теперь слой рендера не протекает в игровую логику.

set_wall_checker(self, fn)
→ None▼

Принимает функцию fn(x: float, y: float) → bool и сохраняет её как self._is_wall. По умолчанию _is_wall = lambda x, y: False — без трассы стен нет. После вызова setup() рендерер подменяет эту лямбду реальной проверкой пикселя в поверхности трассы. Game при этом понятия не имеет, что внутри — это может быть bitmap, полигоны или что угодно.

_is_finish(self, x, y)
→ bool▼

Проверяет, находится ли точка (x, y) в прямоугольной зоне финишной линии. Зона задана константами FINISH_Y=600 (горизонтальная линия), FINISH_X0=1250, FINISH_X1=1550. Допуск по Y: ±30 пикселей. Этот метод передаётся как коллбэк в PlayerState.check_lap().

update(self, dt, a1, a2)
→ None▼

Главный тик игрового мира. Для каждого игрока последовательно: обновляет физику машины (car.update), проверяет коллизию со стеной (_check_wall), проверяет пересечение финиша (ps.check_lap). Порядок важен: сначала физика, потом коррекция позиции.

Engine Engine.py

Оркестратор главного цикла. Единственный класс, который знает о существовании всех трёх компонентов (Game, IRenderer, IInputProvider), но взаимодействует с ними только через интерфейсы. Сам не содержит ни игровой логики, ни рендера — только последовательность вызовов.

__init__(self, game, renderer, input_provider)
→ None▼

Dependency Injection вручную. Принимает зависимости как параметры конструктора — никакого import pygame внутри Engine. Типы параметров — IRenderer и IInputProvider. Это значит Engine можно использовать с любым рендерером (терминал, веб, тесты) без изменения его кода.

run(self)
→ None▼

Запускает бесконечный цикл while True. Каждая итерация состоит из четырёх шагов: 1) poll_events() — события ОС, выход при "quit", сброс при "reset". 2) get_actions() — текущее состояние клавиш. 3) game.update(dt, a1, a2) — физика и логика. 4) render(game) + tick() — рисуем кадр, получаем dt для следующего. Цикл завершается только по команде "quit".

PyGameRenderer Gapi.py

Реализует IRenderer. Отвечает исключительно за визуализацию: создание окна, генерацию геометрии трассы, отрисовку машин, разметки дороги, финишной линии и HUD. Не содержит логики обработки ввода — это отдельный класс.

setup(self, game)
→ None▼

Генерирует поверхность трассы через _build_track(), затем создаёт замыкание is_wall(x, y), которое захватывает surf и offset из локальной области видимости. Это замыкание передаётся в game.set_wall_checker(). Таким образом Game получает способность проверять стены, не имея прямой ссылки на pygame.Surface.

render(self, game)
→ None▼

Отрисовывает кадр в порядке слоёв (back-to-front): 1) Фон. 2) Трасса (blit поверхности со смещением камеры). 3) Пунктирная осевая разметка. 4) Финишная линия (шахматный паттерн из 10 сегментов). 5) Машины (спрайт с вращением по углу). 6) HUD — два полупрозрачных панели с кругами. Камера центрируется на середине между двумя машинами.

_build_track(self)
→ tuple[Surface, tuple]▼

Процедурно генерирует трассу. Вычисляет 72 точки эллипса (шаг 5°, rx=600, ry=300). Рисует трассу в два слоя: сначала широкий серый бордюр (road_w + 40), затем более узкий тёмный асфальт (road_w). Также заполняет self._dashes — список пар точек для пунктирной осевой: каждая чётная пара точек = штрих, нечётная = пропуск. Возвращает поверхность и смещение (min_x, min_y) относительно мировых координат.

PyGameInput Gapi.py

Реализует IInputProvider. Единственная ответственность — считывать ввод с клавиатуры через pygame. Отделён от рендерера в соответствии с принципом единственной ответственности (SRP).

poll_events(self)
→ str▼

Перебирает очередь pygame.event.get(). Событие QUIT (нажатие крестика окна) и клавиша Esc возвращают "quit". Клавиша R возвращает "reset". Если событий нет или они не распознаны — возвращает "continue". Важно: очередь опрашивается ровно один раз за кадр, чтобы не пропускать события.

get_actions(self)
→ tuple[dict, dict]▼

Использует pygame.key.get_pressed() — это мгновенный снимок всех зажатых клавиш, в отличие от poll_events который ловит нажатия/отпускания. Маппинг: Игрок 1 — W/A/S/D + LShift, Игрок 2 — стрелки + Enter (обычный и цифровой).

Gapi Gapi.py

Фасад (паттерн Facade). Не реализует интерфейсов сам — просто создаёт и хранит PyGameRenderer и PyGameInput как публичные атрибуты .renderer и .input. Упрощает точку входа: main.py создаёт один объект и берёт нужные части.

__init__(self)
→ None▼

Создаёт self.renderer = PyGameRenderer() и self.input = PyGameInput(). main.py затем передаёт их раздельно в Engine(game, api.renderer, api.input). Engine принимает типы IRenderer и IInputProvider — полиморфизм обеспечивает, что в будущем можно подменить любой из них независимо.

main.py Точка входа

Точка входа в 4 строки. Выполняет компоновку объектов (composition root): создаёт конкретные реализации, связывает их через Engine и запускает цикл. Блок if __name__ == "__main__" гарантирует, что код не выполнится при импорте файла.

Блок if __name__ == "__main__"
→ выполнение▼

api = Gapi() — инициализирует pygame и создаёт окно. game = Game() — создаёт игровой мир с двумя игроками. Engine(game, api.renderer, api.input).run() — связывает логику с рендером и вводом, запускает бесконечный цикл. Все зависимости инжектируются здесь — ни один класс не создаёт зависимости самостоятельно.

ООП-принципы в проекте

S — Single Responsibility
Каждый класс имеет одну причину для изменения. Car — только физика. PyGameRenderer — только рендер. PyGameInput — только ввод. Раньше Gapi нарушал этот принцип, совмещая оба.
Car: физика\nPyGameRenderer: рендер\nPyGameInput: ввод
O — Open/Closed
Классы открыты для расширения, закрыты для изменения. Чтобы добавить веб-рендерер, создаём новый класс реализующий IRenderer — Engine не трогаем.
class WebRenderer(IRenderer):\n def render(self, game): ...
L — Liskov Substitution
PyGameRenderer и любой другой рендерер взаимозаменяемы везде где ожидается IRenderer. Engine работает одинаково с любой реализацией.
Engine(game, PyGameRenderer(), ...)\nEngine(game, TestRenderer(), ...)
I — Interface Segregation
Два отдельных интерфейса: IRenderer и IInputProvider. Тестовый рендерер не обязан реализовывать ввод. Реализации могут быть полностью независимы.
class MockRenderer(IRenderer):\n # не знает про ввод
D — Dependency Inversion
Engine зависит от абстракций (IRenderer), а не от конкретного PyGameRenderer. Game зависит от коллбэка is_wall(x,y), а не от pygame.Surface.
game.set_wall_checker(fn)\n# не game.track_surf = surf
Composition over Inheritance
PlayerState содержит Car через композицию, а не наследует от него. Game содержит двух PlayerState. Gapi содержит рендерер и ввод.
class PlayerState:\n self.car = Car(x, y)
Защита от копирования
Питонический аналог = delete из C++. Переопределение __copy__ и __deepcopy__ с выбросом исключения блокирует copy.copy() и copy.deepcopy().
def __copy__(self):\n raise TypeError("no copy")
Facade Pattern
Gapi — фасад над двумя подсистемами pygame. main.py создаёт один объект, а не два. При смене платформы меняется только Gapi.
api = Gapi()\nEngine(game, api.renderer, api.input)
Dependency Injection
Все зависимости передаются через конструктор. Engine не делает import pygame и не создаёт Gapi() внутри себя — это делает main.py (Composition Root).
Engine(game, renderer, input)\n# не Engine() { self.gapi = Gapi() }
Callback / Closure
Замыкание is_wall(x,y) захватывает surf и offset из setup(). Game получает поведение без знания о его реализации — инверсия зависимости через функции первого класса.
def is_wall(x, y): # замыкание\n return surf.get_at(...)...
